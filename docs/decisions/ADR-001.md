# ADR-001: Why Ollama Native vs. Containerized

**Status:** Accepted
**Date:** 2026-04-04

## Context

The Mac Studio (128GB Unified Memory, Apple Silicon) serves as the AI inference server for this project. Ollama is the model serving layer. The question: run Ollama natively on macOS, or containerize it in OrbStack alongside everything else?

Every other service in this project runs in OrbStack containers. Running Ollama natively breaks that consistency — it's the one service that lives outside the container stack. This needs justification.

## Decision

**Run Ollama natively on macOS.** Do not containerize it.

### Why

1. **Metal GPU acceleration requires native execution.** Apple's Metal framework — the GPU compute layer — is not available inside Linux containers. OrbStack runs a Linux VM under the hood; containers inside it cannot access the Mac Studio's GPU. Running Ollama natively gives it direct access to the Apple Silicon GPU for inference acceleration.

2. **Unified Memory access requires native execution.** The Mac Studio's 128GB is unified memory — shared between CPU and GPU with no copy overhead. A containerized Ollama would be limited to whatever RAM is allocated to the OrbStack VM, and would not benefit from the zero-copy CPU↔GPU memory architecture. Natively, Ollama can address the full 128GB for loading large models (70B+ parameters).

3. **Model size ceiling matters.** The primary model (Qwen 3.5 122B-A10B) is ~70GB on disk and needs substantial memory when loaded. Constraining Ollama to a container's memory allocation would either prevent loading large models entirely or force aggressive quantization that degrades output quality.

4. **OrbStack GPU passthrough is not mature.** As of early 2026, OrbStack does not support Metal passthrough for containers. Docker Desktop has experimental GPU support on Linux hosts (NVIDIA), but nothing equivalent exists for Apple Silicon Macs.

### Trade-offs accepted

- Ollama is the only service not managed by `docker compose up -d` — it runs as a macOS background process via Homebrew
- Ollama updates are managed separately (`brew upgrade ollama`) rather than by pulling a new container image
- Configuration is via `launchctl setenv` instead of Docker environment variables — a macOS-specific gotcha that's easy to forget after reboots

## Consequences

- Ollama gets full hardware acceleration and full memory access — critical for running 70B+ models at usable speeds
- The container stack (OrbStack) handles everything else: media services, automation, document management
- Network integration is straightforward: Ollama listens on `0.0.0.0:11434` and containers reach it via the host IP
- If Apple or OrbStack ever ships Metal passthrough for containers, this decision can be revisited — but the performance ceiling of native execution will likely always be higher
