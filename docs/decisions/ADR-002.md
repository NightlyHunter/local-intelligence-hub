# ADR-002: Why Qwen 3.5 122B-A10B as Primary Model

**Status:** Accepted (updated from Qwen 2.5 72B)
**Date:** 2026-04-04

## Context

The hub needs a primary LLM for two workloads: interactive chat (via Telegram bot and Open WebUI) and structured output generation (JSON, API calls) for OpenClaw agent tasks and Phase 4 n8n automations. The model must fit in 128GB unified memory and run at acceptable speed on Apple Silicon.

### Previous decision

The original primary model was **Qwen 2.5 72B** (~41GB), chosen for strong structured output, multilingual support, and comfortable fit within 128GB. This was updated when Qwen 3.5 became available on Ollama.

### Models evaluated

| Model | Size on disk | Architecture | Why considered | Verdict |
|---|---|---|---|---|
| Qwen 3.5 122B-A10B | ~70GB | MoE (10B active) | Newer generation, massive context, fast inference despite size | ✅ Primary |
| Qwen 2.5 72B | ~41GB | Dense | Previous primary, proven structured output | Fallback |
| Llama 4 Scout | ~67GB | MoE | Meta's latest, queued for evaluation | Evaluating |
| Nemotron-Cascade-2 | ~18GB | MoE (3B active) | NVIDIA coding/reasoning specialist | Evaluating |
| Llama 4 Maverick | ~245GB | MoE | Too large — exceeds 128GB RAM | ❌ Ruled out |
| DeepSeek V3.2/V4 | >200GB | MoE | Too large — exceeds 128GB RAM | ❌ Ruled out |
| DeepSeek R1 | ~40GB | Dense | Older generation, deprioritized | ❌ Deprioritized |

## Decision

**Qwen 3.5 122B-A10B as the single primary model for all workloads. Qwen 2.5 72B kept as fallback.**

### Why Qwen 3.5 122B-A10B

1. **MoE architecture is the key insight.** Despite being a 122B parameter model, only 10B parameters are active per inference pass. This means response speed is closer to a 10B dense model than a 70B dense model — fast enough to serve as the sole model for both routine queries and heavy tasks.

2. **262K native context window.** Dramatically larger than Qwen 2.5's context, enabling longer documents, multi-turn conversations, and complex agent workflows without truncation.

3. **Newer generation.** Qwen 3.5 represents a full generation improvement over 2.5 in reasoning, instruction following, and structured output — the exact capabilities needed for OpenClaw agent tasks.

4. **Fits in memory.** At ~70GB on disk, it loads comfortably within 128GB unified memory with room for the OS and other processes.

### Why one model, not two

The MoE architecture makes a split strategy (small model for routine, large model for heavy) unnecessary. With only 10B active parameters per pass, the 122B responds fast enough for daily use — there's no need to maintain and swap between a separate lightweight model. Ollama only loads one model at a time anyway, and cold-swapping between models adds latency that defeats the purpose.

### Why Ukrainian language support is not a priority

Initially considered as a model selection factor, but deprioritized. The hub's primary use cases are structured output, agent tasks, and English-language interaction. Multilingual queries are infrequent enough that they don't justify compromising on other capabilities.

## Consequences

- Single model simplifies operations — no need to decide which model to load for which task
- If Qwen 3.5 has Ollama compatibility issues, Qwen 2.5 72B is ready as an immediate fallback with no config changes
- The 122B model has a noticeable cold-start time when loading into memory after being idle
- Model selection will be revisited as new models release — the Ollama ecosystem moves fast, and swapping takes five minutes
