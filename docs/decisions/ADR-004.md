# ADR-004: Remote Access — UniFi Teleport Primary, Cloudflare Tunnel Planned, Tailscale Deferred

**Status:** Accepted (rewrites earlier ADR-004 which proposed Tailscale as the sole remote access method; that decision was never implemented)
**Date:** 2026-04-14 (replaces 2026-03-30)
**Category:** Phase 1 — Network Access

## Context

The hub sits behind T-Mobile home internet, which uses **Carrier-Grade NAT (CGNAT)**. The UDM has no publicly routable IPv4 address, so traditional port forwarding and self-hosted VPN endpoints cannot accept inbound connections from the internet. Any remote access solution has to solve the CGNAT problem structurally — either via a relay/coordination service or via an outbound tunnel.

Three distinct remote access patterns exist in this hub, with different requirements:

- **Pattern A (operator, full access):** The operator, from MacBook or phone, needs broad access to all hub services — Open WebUI, Jellyseerr, Paperless-ngx, arr dashboards, SSH. One trusted human, any protocol.
- **Pattern B (guests, public Jellyfin):** Friends and family streaming Jellyfin from their own devices. They should install nothing, click a URL, and reach only Jellyfin — not admin dashboards, not SSH.
- **Pattern C (machine-to-machine):** The Linux PC (OpenClaw host) reaches the Mac Studio's Ollama endpoint on port 11434. Both machines live on the same LAN; no tunnel is required for this today.

The original ADR-004 proposed **Tailscale** as a single solution for all remote access. Tailscale was never actually deployed — the UniFi ecosystem was found to already provide a workable Pattern A solution, Cloudflare Tunnel is the obvious fit for Pattern B, and Pattern C doesn't need anything. This rewrite documents what's actually in place.

## Decision

Use **three distinct mechanisms** matched to the three patterns, rather than forcing one tool across all of them:

- **Pattern A (operator):** UniFi Teleport / Site Magic, the relay-based service built into UniFi OS.
- **Pattern B (public Jellyfin, Phase 5):** Cloudflare Tunnel. Not implemented yet; planned alongside Phase 5.
- **Pattern C (LAN machine-to-machine):** Plain LAN. No tunnel.

**Tailscale is deferred**, not rejected. It remains a reasonable candidate if a specific triggering need appears: the Linux PC physically moves off the LAN, granular per-device ACLs become necessary, MagicDNS quality-of-life becomes worth the new dependency, or the UniFi side of the stack is retired.

## Rationale

### Why UniFi Teleport for Pattern A

1. **Solves CGNAT without new infrastructure.** Teleport is architecturally a relay service: the UDM establishes an outbound connection to UniFi's coordination backend, and operator devices connect through that relay. No inbound public IP needed. This is the same pattern Tailscale uses (DERP relays + coordination server) — just vendored by UniFi instead.

2. **Reuses infrastructure already in place.** The UDM is already the network's gateway, DHCP server, firewall, and identity boundary. Layering remote access on top of it consolidates management: one app, one set of credentials, one log surface. Adding a separate mesh VPN like Tailscale would mean a parallel identity and ACL system for a problem already solved.

3. **Sufficient for one-operator access.** Pattern A has exactly one user. The features where Tailscale is meaningfully better than UniFi Teleport — fine-grained multi-user ACLs, per-device tagging, SSO integration — are features this hub doesn't need.

4. **No additional third-party dependency.** UniFi OS and its relay infrastructure are already a dependency of the network. Teleport doesn't add one.

### Why Cloudflare Tunnel for Pattern B

1. **No client install for guests.** Friends streaming Jellyfin click a URL and watch. No Tailscale app, no VPN profile, no account creation on an unfamiliar service. For Pattern B, friction is the enemy.

2. **Outbound tunnel sidesteps CGNAT cleanly.** `cloudflared` on the Mac Studio dials out to Cloudflare's edge; no inbound reachability needed on the hub side.

3. **Cloudflare's edge absorbs abuse.** A public Jellyfin URL is a DDoS and probe target. Cloudflare's free tier absorbs the noise before it reaches home infrastructure. Self-hosted public endpoints don't get this for free.

4. **Narrow scope enforced by construction.** A Cloudflare Tunnel exposes one hostname per configured service. `jellyfin.<domain>` reaches port 8096 and nothing else. Admin dashboards stay unreachable because they're not in the tunnel config.

### Why plain LAN for Pattern C

1. **Both endpoints are on the same physical network.** Linux PC and Mac Studio live on the same subnet, behind the same UDM. Ollama is reachable at `http://<mac-studio-lan-ip>:11434` directly. A tunnel would add latency and complexity for no benefit.

2. **No remote-from-the-internet use case.** OpenClaw doesn't need to reach Ollama from outside the network. If it ever does — e.g., if the Linux PC is relocated — that's the trigger to revisit Tailscale or another mesh VPN.

3. **Don't solve problems you don't have.** The cost of the plain-LAN approach is zero; the cost of adding a tunnel is non-zero. The tunnel only pays off when the LAN assumption breaks.

### Why Tailscale was evaluated and deferred

Tailscale is a legitimate alternative to the three-tool approach above. A single Tailscale network could cover Patterns A and C, and Tailscale Funnel could even cover Pattern B. The reasons it's not adopted today:

- **Pattern A is already handled** by infrastructure that exists. Ripping out a working solution to install a functionally equivalent one is churn without upside.
- **Pattern B is better served by Cloudflare** for guest access specifically. Tailscale Funnel works, but creates a public endpoint inside the Tailscale ecosystem rather than leveraging Cloudflare's abuse-absorption.
- **Pattern C doesn't need a solution** until the LAN assumption breaks.
- **Adding Tailscale now would mean three coexisting remote access systems** (UniFi Teleport, Cloudflare, Tailscale) where two suffice. Operational surface area for no concrete gain.

The conditions under which the decision should be revisited:
- Linux PC (or any agent host) moves off the LAN → Pattern C needs a tunnel
- Guest access needs to include more than Jellyfin and a browser-visitable URL isn't enough → Tailscale with ACLs beats per-service Cloudflare Tunnels
- UniFi Teleport is retired or degraded by Ubiquiti → Pattern A needs a new solution and Tailscale is the obvious replacement
- MagicDNS or per-device tagging becomes a meaningful quality-of-life win → subjective, but a valid trigger

## Trade-offs

- **Three systems for three patterns instead of one for all three.** More surface area; three sets of documentation, three failure modes. Accepted because each tool is doing work the others can't do as well.
- **UniFi Teleport is a third-party dependency on Ubiquiti's relay.** Same dependency shape as Tailscale's coordination server. The "self-hosted" framing is equally false for both.
- **Cloudflare Tunnel adds a Cloudflare dependency** for Phase 5. Acceptable for guest-facing service; guest access inherently involves some public endpoint.
- **LAN-only for Pattern C means the Linux PC is geographically pinned.** Moving it off-LAN becomes a work item, not a config change. Accepted until/unless it happens.

## Alternatives Considered

| Option | Why Not |
|---|---|
| Tailscale for everything (original ADR-004) | Never actually implemented; would duplicate Pattern A coverage UniFi already provides; Pattern B better served by Cloudflare |
| Self-hosted WireGuard on a VPS relay | Requires renting and maintaining a VPS; UniFi Teleport provides the same benefit without the VPS |
| Port forwarding | Impossible — T-Mobile CGNAT, no public IP |
| Dynamic DNS + opening router ports | Same CGNAT problem; dynamic DNS doesn't help when there's no public IP to point at |
| ZeroTier | Architecturally similar to Tailscale; same reasoning applies — nothing is broken that ZeroTier fixes |
| Cloudflare Tunnel for operator access too (Pattern A) | Creates public endpoints for admin UIs; even with Cloudflare Access auth, increases attack surface unnecessarily when UniFi Teleport already covers this with no public endpoint |

## Operational Notes

### Pattern A — UniFi Teleport

- Enabled via UniFi Network app → Settings → Teleport VPN (or Site Magic for site-to-site scenarios).
- Operator devices use the WiFiman or UniFi mobile app to initiate connection.
- Teleport links are issued per-device and can be revoked from the UniFi console.
- When connected, the operator device has LAN-equivalent access — reaches all hub services at their LAN IPs (`http://<mac-studio-lan-ip>:3000` for Open WebUI, etc.).

### Pattern B — Cloudflare Tunnel (planned, not yet built)

Outline for when Phase 5 begins:
- `cloudflared` container running on Mac Studio alongside the media stack.
- Tunnel configured with one public hostname per exposed service (starting with Jellyfin).
- Authentication via Cloudflare Access (email-based OTP) to gate even the "public" URL.
- Jellyfin's own authentication stays on as a second layer.
- Per-viewer access control handled in Jellyfin (individual user accounts), not at the tunnel layer.

### Pattern C — LAN

- OpenClaw on the Linux PC points at the Mac Studio's LAN IP for Ollama: `http://<mac-studio-lan-ip>:11434/v1/`.
- If the LAN IP changes (DHCP reassignment), OpenClaw config needs updating. Mitigation: DHCP reservation for the Mac Studio's MAC address in UniFi.
- No tunnel, no overlay, no coordination service involved.

### FileVault note (retained from original ADR-004)

Mac Studio runs with FileVault enabled. Use `sudo fdesetup authrestart` for operator-initiated reboots to avoid post-reboot FileVault lockout. Remote access to a FileVault-locked Mac Studio is impossible regardless of tunnel technology — the OS hasn't finished booting yet. This motivates the disabled-auto-updates policy in ADR-011.
