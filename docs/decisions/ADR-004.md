# ADR-004: Why Tailscale over Traditional VPN / Port Forwarding

**Status:** Accepted
**Date:** 2026-04-04

## Context

The hub needs remote access for several use cases: accessing Open WebUI from a MacBook outside the home, OpenClaw on a separate PC reaching Ollama on the Mac Studio, and eventually sharing Jellyfin with friends (Phase 5). The home internet connection is T-Mobile Home Internet, which uses CGNAT (Carrier-Grade NAT).

### The CGNAT problem

Traditional remote access relies on port forwarding: open a port on your router, point it at an internal service, and connect from outside. CGNAT breaks this entirely. With CGNAT, the ISP assigns a private IP to your router — you don't have a public IP to forward ports on. Your router sits behind another layer of NAT controlled by T-Mobile. No amount of router configuration can fix this because the problem is upstream.

This rules out: direct port forwarding, traditional self-hosted VPN servers (WireGuard, OpenVPN) that need an inbound port, and Dynamic DNS solutions.

## Decision

**Use Tailscale for all remote access.**

### Why

1. **Solves CGNAT with zero configuration.** Tailscale uses WireGuard under the hood but establishes connections via coordination servers and NAT traversal (DERP relays as fallback). Both devices register with Tailscale, and it punches through NAT — including CGNAT — without any port forwarding. It just works.

2. **Mesh VPN, not hub-and-spoke.** Every device on the Tailscale network (tailnet) can reach every other device directly. The Mac Studio, MacBook, Linux PC, and phone all get stable `100.x.x.x` IPs that work from anywhere — home, coffee shop, or cellular.

3. **Zero-trust security model.** Each device authenticates individually. There's no "open port on the internet" attack surface. Services like Ollama, Open WebUI, and Jellyfin are only reachable by authenticated devices on the tailnet — invisible to the public internet.

4. **WireGuard performance.** The underlying WireGuard protocol is fast and lightweight — adds minimal latency. For Jellyfin streaming over Tailscale, the overhead is negligible compared to the stream itself.

5. **Free tier covers this use case.** Tailscale's free plan supports up to 100 devices and 3 users — far more than needed for a personal home lab.

### Alternatives considered

- **Cloudflare Tunnel:** Could expose individual services without port forwarding. But it's HTTP-only (no raw TCP), requires a Cloudflare account, and puts a third party in the data path. Considered as a Phase 5 alternative for friend access (they get a URL instead of installing Tailscale).
- **Self-hosted WireGuard:** The best option if you have a public IP. With CGNAT, it's dead on arrival unless you rent a VPS as a relay — which adds cost and complexity.
- **ZeroTier:** Similar concept to Tailscale. Functional but less polished, smaller community, and Tailscale's macOS integration is superior.

## Consequences

- Every device on the tailnet gets a stable IP that works regardless of physical location or network changes
- Adding a new device takes 30 seconds (install Tailscale, authenticate, done)
- Tailscale is a dependency — if their coordination servers go down, new connections can't be established (existing WireGuard tunnels continue working)
- For Phase 5 (friend sharing), friends either install Tailscale on their device or we use Tailscale Funnel/Cloudflare Tunnel as a zero-install alternative
