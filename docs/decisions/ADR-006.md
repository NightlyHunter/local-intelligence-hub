# ADR-006: Why Tiered Automation Rollout

**Status:** Accepted
**Date:** 2026-04-04

## Context

Phase 4 introduces n8n as the central automation engine, connecting Paperless-ngx, Actual Budget, Ollama, OpenClaw, Jellyfin, and Telegram into automated workflows. The temptation is to wire everything together at once — PDF arrives in Paperless, Ollama extracts the vendor and amount, OpenClaw creates a transaction in Actual Budget, and Telegram confirms it. End-to-end automation from day one.

This is a trap.

## Decision

**Roll out automation in three tiers, advancing only when the previous tier is stable.**

### Tier 1 — Simple notifications (start here)
- New document added to Paperless → Telegram notification
- Jellyfin new media added → Telegram notification
- Daily budget summary from Actual → Telegram

### Tier 2 — AI-assisted processing
- New PDF in Paperless → n8n sends to Ollama for summary → Summary posted to Telegram
- Radarr/Sonarr grab notification → Ollama generates a one-line synopsis → Telegram

### Tier 3 — Full pipeline (only when Tier 1-2 are rock solid)
- New PDF in Paperless → Ollama extracts vendor + amount → OpenClaw creates transaction in Actual Budget → Confirmation via Telegram
- Weekly: OpenClaw queries Actual Budget + Paperless → Generates spending report → Telegram

### Why tiers

1. **Each integration point is a failure point.** A Tier 3 pipeline has 4-5 services in a chain. If Ollama is slow, n8n times out. If Paperless changes its API response format, the extraction breaks. If Actual Budget's API rejects a malformed transaction, you get silent data corruption. Debugging a 5-service chain when you don't know which link broke is miserable.

2. **Tier 1 validates the plumbing.** Simple notifications prove that n8n can talk to each service's API, that webhooks fire correctly, and that Telegram delivery works. These are the same API connections Tier 2 and 3 will use — but with no logic that can break.

3. **Tier 2 validates Ollama integration.** Adding LLM processing introduces latency, prompt engineering challenges, and non-deterministic output. Better to handle these in a low-stakes context (generating a movie synopsis) than in a high-stakes one (creating financial transactions).

4. **Tier 3 is where real damage can happen.** Automated financial transactions with no human in the loop can create incorrect records that compound over time. This tier earns trust by being the last to deploy, after every upstream component has proven reliability.

5. **Limited build time demands incremental progress.** With a full-time job and side gig, available build time is 1-2 hours on weeknights. A tiered approach means each session produces a working, testable result — not a half-built pipeline that doesn't run.

## Consequences

- Automation takes 2-3 months instead of 2-3 weeks — but each step works reliably
- Tier 1 delivers immediate value (notifications) while Tier 3 is still being built
- Each tier's workflows serve as documentation and test cases for the next tier
- If any tier proves unstable, the tiers below it continue functioning independently
- The temptation to skip tiers will be strong — resist it
