# ADR-008: Why gluetun VPN Kill Switch for Torrent Traffic

**Status:** Accepted
**Date:** 2026-04-04

## Context

The media stack downloads content via torrents through qBittorrent. Torrent traffic exposes the downloader's IP address to every peer in the swarm. Without a VPN, the home IP is visible to anyone monitoring public trackers — including copyright enforcement agencies. A VPN masks the real IP, but if the VPN connection drops (even briefly), qBittorrent falls back to the regular internet connection and the real IP leaks.

The goal: route all torrent traffic through a VPN, and if the VPN drops, all torrent traffic stops completely. Zero leaks, ever.

## Decision

**Use gluetun as a VPN container with qBittorrent sharing its network namespace. ProtonVPN with WireGuard protocol.**

### How the kill switch works

```
qBittorrent → (network_mode: service:gluetun) → gluetun → ProtonVPN WireGuard → Internet
```

qBittorrent has no network interface of its own. It uses `network_mode: "service:gluetun"` in Docker Compose, which means it shares gluetun's network namespace entirely. If gluetun's VPN tunnel drops, qBittorrent has no internet at all — not degraded, not falling back to the host network, just nothing. This is the kill switch.

qBittorrent's ports (WebUI on 8080, torrents on 6881) are defined on the gluetun container, not on qBittorrent, because gluetun owns the network.

### Why ProtonVPN + WireGuard

1. **ProtonVPN** is a reputable no-logs provider with a track record of defending user privacy. Switzerland-based, outside Five Eyes/Fourteen Eyes jurisdictions.

2. **WireGuard over OpenVPN** was a learned decision. The initial setup used OpenVPN, which failed due to DNS resolution timeouts inside the tunnel and route conflicts with OrbStack's networking. The tunnel would connect (handshake succeeded) but couldn't route traffic — healthchecks timed out, and gluetun restarted in a loop. Switching to WireGuard resolved all networking issues immediately — cleaner routing, lower overhead, faster connection establishment.

3. **Netherlands server** chosen for proximity to well-seeded European torrent swarms and ProtonVPN's strong Netherlands infrastructure.

### Lessons learned during setup

- **OpenVPN + OrbStack networking don't play well together.** OpenVPN's TUN device and route management conflicted with OrbStack's virtual networking. Gluetun connected to ProtonVPN successfully, but DNS queries timed out and the healthcheck loop killed the container repeatedly. Hours of debugging DNS settings (`DOT=off`, `DNS_ADDRESS=1.1.1.1`) didn't resolve it.
- **WireGuard just worked.** Switching `VPN_TYPE` from `openvpn` to `wireguard` and generating a WireGuard key from ProtonVPN's web panel fixed everything in one restart.
- **ProtonVPN WireGuard key generation:** Account portal → WireGuard config → Platform: Linux, VPN Accelerator: on, all other options off. The private key goes in `.env`, never in the compose file.
- **DHT bootstrapping requires patience.** After a container restart, qBittorrent's DHT node count starts at 0 and takes 2-3 minutes to bootstrap through the VPN tunnel. Magnet links won't resolve metadata until DHT is populated. If magnets are stuck in "downloading metadata" state, restart gluetun and wait.
- **FlareSolverr is needed for Cloudflare-protected indexers.** 1337x and EZTV sit behind Cloudflare challenges. FlareSolverr solves these in ~11 seconds per challenge, configured per-indexer in Prowlarr (not globally).
- **1337x magnet links > torrent files.** Torrent file downloads from 1337x timed out even with FlareSolverr. Switching Prowlarr to prefer magnet links for 1337x bypasses the file download entirely.

### Alternatives considered

- **qBittorrent's built-in proxy support:** Supports SOCKS5 proxies, but proxy connections can drop silently without triggering a kill switch. The `network_mode` approach is more robust because the kill switch is at the network layer, not the application layer.
- **Running VPN on the host:** Would route ALL traffic through the VPN, including services that don't need it (Jellyfin, Prowlarr, Open WebUI). Container-level VPN is surgical — only qBittorrent traffic is routed.
- **Mullvad:** Another strong no-logs provider. ProtonVPN was already available, so no reason to add another subscription.

## Consequences

- All torrent traffic is invisible to the ISP — they see a WireGuard connection to a ProtonVPN server, nothing else
- If ProtonVPN's Netherlands servers go down, all downloads stop until the tunnel reconnects (acceptable — no data loss, just a pause)
- qBittorrent WebUI is accessed via gluetun's port mapping (`localhost:8080`), which can be confusing when debugging — remember that ports belong to gluetun, not qBittorrent
- The `.env` file containing the WireGuard private key is `.gitignore`'d and never committed
