# ADR-008: gluetun VPN Kill Switch for All Torrent Traffic

**Status:** Accepted
**Date:** 2026-04-04
**Category:** Phase 2 — Media Stack

## Context

The media stack downloads torrents via qBittorrent. Torrent traffic exposes the source IP to every peer in the swarm. Without a VPN, the Mac Studio's real IP is visible to anyone monitoring public torrent trackers — including copyright enforcement agencies.

A VPN solves this, but a naive VPN setup has a critical failure mode: if the VPN connection drops, the torrent client falls back to the default network interface and continues downloading over the unprotected connection. This "VPN leak" can happen silently during a tunnel renegotiation, a provider outage, or a network change.

## Decision

Route all torrent traffic through **gluetun** (VPN container) using Docker's `network_mode: "service:gluetun"` on the qBittorrent container, creating a **network-level kill switch**.

## Rationale

1. **Network-level isolation, not application-level.** qBittorrent uses `network_mode: "service:gluetun"`, meaning qBittorrent has no network interface of its own — it shares gluetun's network stack entirely. If gluetun's VPN tunnel drops, qBittorrent has literally no network path to the internet. This is enforced by Docker networking, not by qBittorrent's settings or a firewall rule that might be misconfigured.

2. **Zero-leak guarantee.** Unlike application-level kill switches (qBittorrent's built-in "Network interface" binding), the container-level approach cannot leak under any circumstance. There is no race condition between tunnel drop and kill switch activation — without gluetun's tunnel, qBittorrent has no connectivity at all.

3. **VPN provider independence.** gluetun supports 60+ VPN providers and both WireGuard and OpenVPN protocols. Switching providers or protocols requires changing environment variables, not rebuilding the architecture. Currently running ProtonVPN with WireGuard (Netherlands exit).

4. **Other services unaffected.** Only qBittorrent routes through gluetun. Radarr, Sonarr, Prowlarr, Jellyfin, and all other containers use the default Docker bridge network. A VPN outage affects only downloads, not browsing, streaming, or API communication.

5. **Port management centralized.** Since qBittorrent shares gluetun's network, qBittorrent's web UI port (8080) is exposed through gluetun's port mapping. This keeps all network configuration for the torrent pipeline in one place.

## Trade-offs

- **gluetun restart kills qBittorrent networking.** If gluetun restarts (e.g., Watchtower updates the image), qBittorrent loses connectivity until gluetun is back up. Active downloads pause and resume automatically. This is the intended behavior — brief downtime is preferable to any leak window.
- **Debugging complexity.** Network issues in qBittorrent require inspecting gluetun's logs first, since qBittorrent has no independent network. Adds one layer of indirection to troubleshooting.
- **OpenVPN failed initially.** OpenVPN had DNS resolution timeouts inside the tunnel and route conflicts with OrbStack's virtual networking. WireGuard worked immediately — kept as the active protocol.

## Architecture

```
Internet ← WireGuard tunnel ← gluetun container ← qBittorrent (shared network)
                                    ↑
                               ProtonVPN (Netherlands)

If tunnel drops:
Internet ✗ (no route) ✗ gluetun container ← qBittorrent (no connectivity)
```

All other containers (Radarr, Sonarr, Jellyfin, etc.) use the default OrbStack bridge network and are unaffected by VPN state.

## Alternatives Considered

| Option | Why Rejected |
|---|---|
| qBittorrent built-in network binding | Application-level — race condition possible during tunnel drops; relies on correct interface name configuration |
| Host-level VPN (entire Mac Studio) | Routes all traffic through VPN including Ollama API calls, remote access, and Jellyfin streaming — unacceptable side effects |
| iptables firewall rules on qBittorrent | Requires privileged container, manual rule management, and can be misconfigured; Docker network isolation is more robust |
| Separate VPN-routed VM | Over-engineered for a home lab; gluetun achieves the same isolation with a single container |

## Operational Notes

- ProtonVPN WireGuard key generated via `account.protonvpn.com` → Linux platform, VPN Accelerator on, all other options off.
- Verify VPN is active: `docker exec gluetun wget -qO- ifconfig.me` should return a Netherlands IP.
- gluetun health check configured — container reports unhealthy if tunnel is down.
- qBittorrent limits: 3 active downloads, 5 active uploads, 8 active torrents (conservative for 1TB SSD; increase after NAS).
